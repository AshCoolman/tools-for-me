# @ashcoolman/leaf-toolkit

Coverage-driven work loops for monorepos. Partition a codebase into
bite-sized **leaves**, rank them by priority, then drive an AI agent loop
that picks the highest-priority leaf below target and writes tests.

A plugin contract is sketched for other domains (refactor, security,
docs) but **only the coverage domain is wired today** — see Status.

## Status

- **Native ports complete.** All built-in verbs are implemented:
  `partition`, `priority`, `survey`, `link coverage`, `status coverage`,
  `scope-from-priority`, `safe-tool`, `safe-vitest`, `domain register`.
- **Coverage is the only domain.** The plugin contract exists; registry
  dispatch isn't wired yet — `leaf link` and `leaf status` currently
  hardcode `coverage`. Other domains (refactor / security / docs) are
  not usable until the dispatcher lands.
- **Test runner.** `leaf survey` only knows **vitest** and **jest**. No
  mocha / ava / tap / pytest support today.
- **Repo shape.** `leaf link coverage` resolves a leaf to its owning
  workspace via the root `package.json` `workspaces` field, so this
  toolkit assumes a **yarn 1 / yarn berry / npm workspaces / pnpm
  workspaces** monorepo. Single-package repos are not supported yet.

## Requirements

- Node `>=20`.
- A monorepo with `workspaces` declared in root `package.json`.
- `vitest` or `jest` installed in every workspace you want surveyed.
- `pgrep` (default `safe-tool` backend, present on every macOS / Linux).
- Optional: `flock` from `util-linux` for the semaphore backend.

## Filename convention

Generated artefacts use a `*.gitignored*` infix so a single
`*.gitignored*` glob in your `.gitignore` covers them:

- `leaves.gitignored.json` — partition manifest (root)
- `leaf-coverage-scope.gitignored.json` — derived exclude list (root)
- `coverage-survey.gitignored/<pkg>.txt` — per-package coverage output
- `LEAF.partition[.bin-N].md`, `LEAF.coverage[.bin-N].md`,
  `LEAF.audit[.bin-N].md` — per-leaf, regenerated each run

`LEAF.priority[.bin-N].md` is the only LEAF doc you commit — it carries
durable priority decisions. Add this to `.gitignore`:

```
*.gitignored*
LEAF.partition*.md
LEAF.coverage*.md
LEAF.audit*.md
```

If you don't like the convention, the output filenames are not
configurable today — pin to a fork or open an issue.

## Concepts

### Leaf

A small subtree of the repo (~1000 LOC) that groups files which sensibly
travel together. The toolkit auto-partitions, with bins for oversize
trees.

### LEAF docs

Each leaf has one or more `LEAF.<domain>.<ext>` docs alongside its
source. Each domain owns its own file so concerns stay isolated:

- `LEAF.partition.md` — files in the leaf, scope (subtree/bin), neighbours. Generated.
- `LEAF.priority.md` — priority field (p0–p5) + rationale. Set by humans, durable.
- `LEAF.coverage.md` — coverage status + blockers. Generated by status linker.
- `LEAF.audit.md` — risky logic / volatility / important code. Free-form, sub-agent owned.
- `LEAF.<anything>.<ext>` — register a custom work-type plugin and emit your own.

Each file has YAML frontmatter (machine-readable) plus markdown body
(LLM-readable + human-readable). Hybrid is deliberate: parsers read
frontmatter, agents read everything.

Bin-scoped leaves disambiguate by index: `LEAF.coverage.bin-2.md`.

### Priority dictates scope

The leaf priority field is the **only** lever for "is this code in scope
for X?" Coverage thresholds, refactor cycles, security reviews — all
derive their candidate list from priority. Never hand-edit a tool's
exclude list to make a metric look better; raise (or accept) the
priority instead.

This rule is enforced by `leaf scope-from-priority`, which generates a
`leaf-coverage-scope.gitignored.json` that downstream configs (vitest,
eslint, sonar, etc.) consume.

### safe-tool: the flock principle

Multiple parallel agents calling the same heavy tool (`vitest`, `tsc`,
`playwright`) can saturate RAM. `leaf safe-tool` wraps any command,
counts how many sibling instances are already running, and blocks until
under a configurable cap. Two backends ship:

- **pgrep poll** (default, portable): counts matching procs every N seconds.
- **flock semaphore** (linux/macOS with `util-linux` flock): a real
  counted semaphore via N file locks; no busy-poll.

Use `--flock-dir <path>` to opt into the flock backend.

## CLI surface

```sh
leaf partition                       # write leaves manifest + LEAF.partition.md per leaf
leaf priority [--distribute]         # interactive TUI to assign p0..p5; --distribute prints histogram + reclassify
leaf survey [--only=<substr>]        # run per-workspace coverage → coverage-survey.gitignored/<pkg>.txt
leaf link coverage                   # write LEAF.coverage.md per leaf from survey output
leaf status coverage --target N --metric all|lines|branches|funcs|stmts [--below-target] [--json]
leaf scope-from-priority             # emit JSON of files in low/lowest priority leaves
leaf safe-tool --cap N --match P -- <cmd…>   # concurrency-capped wrapper (pgrep poll or --flock-dir semaphore)
```

The toolkit resolves the consumer's repo root via `LEAF_REPO_ROOT`
(preferred) or by walking up from cwd looking for a
`leaves.gitignored.json` / `package.json`.

### `--distribute` target curve

The default curve `leaf priority --distribute` reclassifies against is
opinionated:

| Priority | Target share |
| --- | --- |
| p0 | 5%  |
| p1 | 8%  |
| p2 | 17% |
| p3 | 25% |
| p4 | 30% |
| p5 | 15% |

Roughly Pareto-shaped — most code in p3/p4, a thin top in p0/p1. It is
**not configurable in the CLI today**; treat it as a sane default and
fork the source (`src/commands/priority.ts`) if your project's shape is
materially different.

## LEAF.<domain>.md format

```md
---
domain: coverage
leafPath: packages/foo/src
generatedAt: 2026-04-29T08:00:00Z
metric: vitest
target: 95
status:
  lines: 93.96
  branches: 96.71
  funcs: 98.21
  stmts: 93.96
---

# Coverage — packages/foo/src

## Files
- packages/foo/src/Bar.tsx
- packages/foo/src/baz.ts

## Blockers
- Bar.tsx:82-93 dead code `yToIdx`. Recommend deletion.
```

## Plugin contract (work-type)

A domain plugin is a TS module exporting:

```ts
export interface DomainPlugin<Status> {
  name: string;                                    // e.g. "coverage"
  fetchStatus(leaf: Leaf): Promise<Status>;        // current state
  isAtTarget(status: Status, target: unknown): boolean;
  renderDoc(leaf: Leaf, status: Status): {         // LEAF.<name>.md body
    frontmatter: Record<string, unknown>;
    body: string;
  };
  scopeOptOut?(leaf: Leaf, priority: Priority): string[] | null;
}
```

Builtins ship for `partition`, `priority`, `coverage`. Add more via
`leaf domain register` — note the registry dispatcher isn't wired yet,
so registered domains aren't usable from `leaf link` / `leaf status`
until that lands.

### Installing the skills (Claude Code consumers)

```bash
cp -r node_modules/@ashcoolman/leaf-toolkit/skills/* ~/.claude/skills/
```

Or symlink for live updates:

```bash
ln -s "$PWD/node_modules/@ashcoolman/leaf-toolkit/skills/code-leaf-add-coverage" \
      ~/.claude/skills/code-leaf-add-coverage
ln -s "$PWD/node_modules/@ashcoolman/leaf-toolkit/skills/code-leaf-link-coverage" \
      ~/.claude/skills/code-leaf-link-coverage
```

The skills invoke `yarn leaf <verb>` so they work in any yarn-based
project that has the package installed. For npm/pnpm consumers, swap
the prefix in your local copy.

## Consumer wiring (host repo)

After `npm i -D @ashcoolman/leaf-toolkit`:

1. Add the entries from the **Filename convention** section above to
   `.gitignore` (commit `LEAF.priority.md` — it captures durable
   decisions).
2. In your test runner config, read `leaf-coverage-scope.gitignored.json`
   and spread its `excludeFiles` into `coverage.exclude`. Example for
   vitest:

   ```ts
   // vite.config.ts
   import { existsSync, readFileSync } from "node:fs";
   const scopePath = "leaf-coverage-scope.gitignored.json";
   const lowPri = existsSync(scopePath)
     ? (JSON.parse(readFileSync(scopePath, "utf8")) as { excludeFiles?: string[] })
         .excludeFiles ?? []
     : [];
   export default defineConfig({
     test: {
       coverage: {
         include: ["src/**/*.{ts,tsx}"],
         exclude: ["src/**/*.test.{ts,tsx}", ...lowPri],
       },
     },
   });
   ```
3. Run `leaf partition` → `leaf priority` → `leaf survey` → `leaf link coverage`.
   From here every loop iteration is a single `leaf status` query plus
   an optional skill.
