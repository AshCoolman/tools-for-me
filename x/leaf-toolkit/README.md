# @ashcoolman/leaf-toolkit

Partition a codebase into bite-sized leaves, rank them by priority, and drive
work loops (test coverage, refactors, security review, doc debt — anything that
benefits from per-area ranking) against them.

> **Status:** scaffold. Will be extracted to its own repo soon.

## Concepts

### Leaf
A small subtree of the repo (~1000 LOC) that groups files which sensibly travel
together. The toolkit auto-partitions, with bins for oversize trees.

### LEAF docs
Each leaf has one or more `LEAF.<domain>.<ext>` docs alongside its source. Each
domain owns its own file so concerns stay isolated:

- `LEAF.partition.md` — files in the leaf, scope (subtree/bin), neighbors. Generated.
- `LEAF.priority.md` — priority field (p0–p5) + rationale. Set by humans, durable.
- `LEAF.coverage.md` — coverage status + blockers. Generated by status linker.
- `LEAF.refactor.md` — refactor TODOs (your domain).
- `LEAF.security.md` — security findings (your domain).
- `LEAF.<anything>.<ext>` — register a custom work-type plugin and emit your own.

Each file has YAML frontmatter (machine-readable) plus markdown body
(LLM-readable + human-readable). Hybrid is deliberate: parsers read frontmatter,
agents read everything.

Bin-scoped leaves disambiguate by index: `LEAF.coverage.bin-2.md`.

### Priority dictates scope
The leaf priority field is the **only** lever for "is this code in scope for X?"
Coverage thresholds, refactor cycles, security reviews — all derive their
candidate list from priority. Never hand-edit a tool's exclude list to make a
metric look better; raise (or accept) the priority instead.

This rule is enforced by `leaf scope-from-priority`, which generates a
`leaf-scope.gitignored.json` (or domain-equivalent) that downstream configs
(vitest, eslint, sonar, etc.) consume.

### safe-tool: the flock principle
Multiple parallel agents calling the same heavy tool (`vitest`, `tsc`,
`playwright`) can saturate RAM. `leaf safe-tool` wraps any command, counts how
many sibling instances are already running, and blocks until under a configurable
cap. Two implementations ship:

- **pgrep poll** (default, portable): counts matching procs every N seconds.
- **flock semaphore** (linux/macOS with `util-linux` or `procmail`): a real
  counted semaphore via N file locks; no busy-poll.

Use `--flock-dir <path>` to opt into the flock backend.

## CLI surface

```sh
leaf partition                 # generate leaves.gitignored.json + LEAF.partition.md per leaf
leaf priority                  # interactive priority TUI; updates LEAF.priority.md
leaf link <domain>             # update LEAF.<domain>.md from <domain>'s status source
leaf status <domain> --target  # query leaves below target
leaf scope-from-priority       # emit JSON of files in low/lowest leaves
leaf safe-tool -- <cmd…>       # run cmd under concurrency cap
leaf domain register <name>    # scaffold a new work-type plugin
```

## LEAF.<domain>.md format

```md
---
domain: coverage
leafPath: packages/foo/src
generatedAt: 2026-04-29T08:00:00Z
metric: vitest
target: 95
status:
  lines: 93.96
  branches: 96.71
  funcs: 98.21
  stmts: 93.96
---

# Coverage — packages/foo/src

## Files
- packages/foo/src/Bar.tsx
- packages/foo/src/baz.ts

## Blockers
- Bar.tsx:82-93 dead code `yToIdx`. Recommend deletion.
```

## Plugin contract (work-type)

A domain plugin is a TS module exporting:

```ts
export interface DomainPlugin<Status> {
  name: string;                                    // e.g. "coverage"
  fetchStatus(leaf: Leaf): Promise<Status>;        // current state
  isAtTarget(status: Status, target: unknown): boolean;
  renderDoc(leaf: Leaf, status: Status): {         // LEAF.<name>.md body
    frontmatter: Record<string, unknown>;
    body: string;
  };
  scopeOptOut?(leaf: Leaf, priority: Priority): string[] | null;
}
```

Builtins ship for `partition`, `priority`, `coverage`. Add more via
`leaf domain register`.

## Layered packaging plan

- **Layer 1** (this package): framework-agnostic CLI + plugin contract.
- **Layer 2** (`@ashcoolman/leaf-toolkit-claude`): Claude Code plugin bundling
  skills + memory rules + agent prompt templates that drive the loop.
- **Layer 3** (speckit pack): constitution + spec/plan templates that
  reference leaves.

Layer 1 stands alone — non-Claude teams use it with their own runner.
